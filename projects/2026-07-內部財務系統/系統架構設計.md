# 內部財務系統 — 架構設計

> 最後更新:2026/07/13
> 狀態:設計階段,尚未選定工具、尚未建置

## 背景與範圍

Jinni 手上同時管理三筆帳:共學團(伊利歐)、職人方舟工作坊、方舟定向課程(獨立於方舟工作坊之外的課程)。基金會有專職會計處理正式帳務,但 Jinni 需要一套內部系統做三件事:

1. 隨時清楚每個學生/活動誰付了、誰沒付
2. 學生繳費後可直接更新資料,行政依此處理後續入帳與收據,不用重複對帳
3. 掌握所有活動、課程的成本結構,不只看收入

金流路徑已確認:三筆帳的收費都經過基金會(旅學項目除外,依 `context/foundation.md` 規則)。這份文件只設計系統架構,不涉及實際金流異動。

## 資料結構設計

系統需要四張互相關聯的表,不管最後落在 Notion 還是試算表,這個結構都成立。

### 1. 學生/家長主檔(Roster)

跨三筆帳共用一份主檔,同一個學生若同時在共學團與方舟出現,不用重複建檔。

| 欄位 | 說明 |
|---|---|
| 學生姓名 | |
| 所屬帳(可複選) | 共學團 / 職人方舟工作坊 / 方舟定向課程 |
| 家長/監護人聯絡方式 | 電話、Line、Email |
| 學生聯絡方式 | 若學生自己對接 |
| 加入時間 | |
| 備註 | 特殊付款約定、分期等 |

### 2. 活動/課程主檔(Programs)

每學期、每場次建一筆,方舟工作坊一年 10 場、三季制,這裡就是拆場次的地方。

| 欄位 | 說明 |
|---|---|
| 活動名稱 | |
| 所屬帳 | 共學團 / 職人方舟工作坊 / 方舟定向課程 |
| 學期/季別 | 見下方「時間軸定義」統一命名慣例 |
| 起訖日期 | 實際起訖日,篩選與自動歸類都靠這欄,不靠學期文字 |
| 金流路徑 | 經基金會 / 不經基金會(旅學) |
| 每人應收金額 | |
| 預計人數 | |
| 預算總額 | 用於成本結構比對 |
| 狀態 | 籌備中 / 報名中 / 進行中 / 已結束 |

### 3. 繳費紀錄(Payments)— 核心表

每個學生對每個活動一筆,學生更新的就是這張表。

| 欄位 | 說明 |
|---|---|
| 學生 | 連結學生主檔 |
| 活動 | 連結活動主檔 |
| 應繳金額 | |
| 已繳金額 | |
| 繳費狀態 | 未繳 / 部分繳納 / 已繳清 |
| 繳費方式 | 匯款 / 現金 / 其他 |
| 繳費日期 | |
| 入帳狀態 | 待入帳 / 已入帳(行政更新) |
| 收據狀態 | 待開立 / 已開立(行政更新) |
| 備註 | |

「誰付了、誰沒付」這個問題,就是這張表按活動篩選、看繳費狀態欄位。

### 4. 成本項目(Cost Items)

每個活動結束後,或籌備期就先估,填實際支出,用來看成本結構。

| 欄位 | 說明 |
|---|---|
| 活動 | 連結活動主檔 |
| 項目名稱 | 例如場地費、講師費、材料費 |
| 類別 | 場地 / 講師 / 材料 / 行政 / 其他 |
| 預算金額 | |
| 實際支出 | |
| 備註 | |

活動主檔可以做一個「收入加總 − 成本加總」的自動計算欄位,這樣每個活動的損益一眼就看得到,三筆帳橫向比較也方便。

## 流程設計

**學生/家長端**
匯款後填寫 Google 表單回報金額、日期、對應活動。家長/學生完全不進系統後台,表單只能新增一筆提交紀錄,不能查看或修改任何既有資料——這樣不會有人誤改到別人的繳費紀錄,也不會看到別人的繳費狀況或成本資料。

表單資料進到後台繳費紀錄表的方式有兩種,先列出來,實際落地時再定:
- **人工登打**:行政每天或每收到通知就把表單新回覆謄進繳費紀錄表,簡單但多一道手動步驟
- **自動同步**:比照方舟現有「家長確認信用 Google Apps Script 自動化」的做法,寫一支 Apps Script 把表單新回覆自動寫進後台(Notion 或 Sheet),行政只需要事後核對,不用手動謄打

**Jinni 端**
隨時看總表,依帳別、活動篩選「誰沒付」;活動結束後看成本項目表,掌握損益。

**行政端**
看「入帳狀態 = 待入帳」的清單 → 對照基金會流程執行正式入帳 → 回填「入帳狀態 = 已入帳」→ 開立收據 → 回填「收據狀態 = 已開立」。行政不用再自己對帳,系統已經告訴他們誰繳了錢在等處理。

## 權限設計

- **Jinni**:全部可讀,可編輯活動主檔與成本項目
- **行政**:可讀繳費紀錄,可更新入帳狀態、收據狀態;不動成本項目與活動預算(視需要調整)
- **學生/家長**:完全沒有後台存取權限,只能透過 Google 表單提交新的繳費回報,看不到其他人的資料,看不到成本項目,也看不到自己以外任何既有紀錄

只有團隊(Jinni + 行政)能進後台這件事是硬性的,家長端一律走表單,不開放任何形式的後台帳號或分享連結。

## 三筆帳共用 vs 各自獨立

學生主檔建議共用一份,用「所屬帳」欄位做標籤區分,避免同一個學生在共學團跟方舟各建一次檔。活動主檔、繳費紀錄、成本項目則天生就會按活動/場次分開,不需要額外切三份系統,篩選「所屬帳」欄位就能各自看各自的。

## 時間軸定義

三筆帳的季別制不同,不能共用同一套「第幾季」的說法,否則行政跟 Jinni 對到「Q1」會想到不同的月份範圍。

| 帳別 | 制度 | Q1 | Q2 | Q3 |
|---|---|---|---|---|
| 職人方舟工作坊 | 當年度制(西元年) | 1–4 月 | 5–8 月 | 9–12 月 |
| 共學團 | 學年制 | 9–12 月 | 隔年 1–4 月 | 接續 5–8 月 |
| 方舟定向課程 | 學年制 | 9–12 月 | 隔年 1–4 月 | 接續 5–8 月 |

這也是為什麼活動主檔要有「起訖日期」這個實際日期欄位,不能只靠「學期/季別」這種文字標籤——工作坊的 Q1 跟共學團的 Q1 是完全不同的月份,篩選跟報表都要靠起訖日期去對,才不會混淆。

命名慣例建議:

- 工作坊:用「西元年-Q 數」,例如 `2026-Q1`(2026 年 1–4 月)
- 共學團 / 定向課程:用「學年起始年-Q 數」,例如 `2026 學年-Q1`(2026 年 9 月–12 月)、`2026 學年-Q2`(2027 年 1–4 月)

這樣光看標籤就知道是哪個帳別的季度,不用再對照月份。

## 工具選擇的考量(先不決定)

你提到每年、每學期都要更新,不確定哪種工具比較適合,這裡先列考量點,之後再一起拍板:

**Notion**
方舟工作坊已經有報名收費資料庫(SOP v1.1),學習成本低。Notion 的資料庫之間可以互相連結(學生連活動連繳費紀錄),「誰沒付」這種跨表篩選查詢做得到,而且可以做範本(template),每學期用範本快速建立新一批活動與繳費紀錄,結構不用重畫。缺點是如果之後要改欄位或公式,舊資料要小心遷移,且行政與學生若要各自填寫,需要設定好權限與表單入口。

**Excel / Google Sheet**
上手快,Jinni 跟行政應該都熟悉。但四張表之間的關聯(誰在哪個活動、成本自動加總到活動損益)要靠公式或樞紐分析表手動維護,學期一多容易變成好幾份檔案對不起來,「誰沒付」這種即時查詢會比較費工。

**傾向**:如果這套系統要長期用、且核心價值是「即時知道誰沒付」跟「成本結構橫向比較」,Notion 的關聯資料庫結構比較貼近需求,方舟也已經有基礎。但這不是定案,實際落地前建議先用這份架構在 Notion 或 Sheet 各拉一個小範本(例如只放一個活動、五個學生)試跑一輪,感受一下操作順不順手,再決定要不要全面搬。

## 待確認事項

- 收據是否由基金會統一開立,還是方舟工作坊、共學團、定向課程各自有窗口
- 方舟定向課程目前是否已經有報名機制(表單、Line 群組等),還是這次要一併從零設計
- 學生是否可能同時橫跨兩筆以上的帳(例如同時在共學團與方舟工作坊),這會影響 Roster 表「所屬帳」欄位要不要設計成複選
- 行政更新入帳/收據狀態的頻率與觸發方式(每天巡一次待入帳清單,還是收到通知才處理)
